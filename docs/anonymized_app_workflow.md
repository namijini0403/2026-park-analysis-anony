# 비식별 앱 단계별 작업 절차

## 현재 상태 확인

- 기존 식별 가능 앱 저장소: `../2026-park-analysis`
- 비식별 제출 앱 저장소: 현재 폴더
- 두 저장소는 Git 루트가 다르며, 제출 앱은 별도 주소로 배포한다.
- 현재 제출 앱은 v0 골격이다. 최종 목표는 기존 앱의 디자인과 기능 구조를 최대한 재현하면서 식별자만 제거하는 것이다.

## 단계 0. 보호선 고정

목표:
- 기존 앱을 수정하지 않는다.
- 새 저장소에서만 작업한다.
- 공개 금지 파일이 Git 추적 대상에 들어오지 않게 한다.

작업:
- `.gitignore` 점검
- `npm run check:safety` 점검 범위를 `src`, `public`, `dist`, `docs`, 주요 설정 파일로 유지
- `public/data/*.json` 외 데이터 산출물을 커밋하지 않기

완료 기준:
- 기존 앱 저장소의 tracked diff가 없다.
- 새 저장소에서 원본 CSV, GeoJSON, XLSX가 Git 추적 대상이 아니다.

성공 검증:
- 기존 앱 저장소에서 `git diff --stat` 결과가 비어 있어야 한다.
- 기존 앱 저장소에서 `git status --short`를 보고, 이번 작업으로 생긴 tracked 수정이 없어야 한다.
- 비식별 앱 저장소에서 `git rev-parse --show-toplevel` 결과가 현재 폴더여야 한다.
- 비식별 앱 저장소에서 `git status --short --ignored`를 확인해 `dist/`, `node_modules/`, 로그, tsbuildinfo가 ignore 처리되어야 한다.
- `npm run check:safety`가 통과해야 한다.

## 단계 1. 기존 앱 구조 대응표 작성

목표:
- 기존 앱의 첫 화면, 지도, 상세 리포트, 시뮬레이션, 전체 통계, 앱 요약 기능을 비식별 앱의 화면 단위로 대응시킨다.

작업:
- 기존 앱의 주요 화면과 컴포넌트 역할을 문서화한다.
- 각 화면별로 유지할 기능과 제거할 식별자를 나눈다.
- 지도는 실제 지도 대신 도식지도 컴포넌트로 대응한다.

완료 기준:
- 화면별 “유지 기능”과 “비식별 대체 방식”이 명확하다.

성공 검증:
- `docs/original_app_mapping.md`를 작성해 기존 앱의 화면 단위와 비식별 앱 대응 화면을 1:1로 정리한다.
- 대응표에는 최소한 첫 화면, 지도, 학교 패널, 상세 리포트, 시뮬레이션, 전체 통계, 앱 요약/가이드가 포함되어야 한다.
- 각 화면마다 “유지할 기능”, “제거할 식별자”, “대체 표현”, “필요 데이터 JSON” 칸이 있어야 한다.
- 문서 안에 원본 학교명, 학교ID, 실제 좌표, 실제 시설명이 들어가면 안 된다.
- `npm run check:safety`가 통과해야 한다.

## 단계 2. 비식별 데이터 모델 확장

목표:
- 기존 앱 기능을 재현할 수 있을 만큼 제출용 JSON을 확장한다.

작업:
- 학교 목록, 전체 통계, 상세 리포트, 시뮬레이션, 앱 요약에 필요한 JSON 구조를 분리한다.
- 학교 중심 도식지도용 데이터 추가:
  - 반경 500m
  - 도보 500m 범위의 추상 polygon 또는 단순화된 shape
  - 공원 상대 위치
  - 후보지 상대 위치
  - 단절요소 상대 위치
- 실제 좌표는 출력하지 않는다.
- 상대좌표는 학교 중심 변환 후 고정 난수 회전/반전을 적용한다.

완료 기준:
- `public/data/*.json`만으로 앱 주요 화면이 작동한다.
- 안전 스캔에서 원본 식별자 패턴이 검출되지 않는다.

성공 검증:
- `npm run build:data`가 실패 없이 끝나야 한다.
- `public/data/manifest.json`에 생성 시각과 산출 JSON 목록이 기록되어야 한다.
- `public/data/*.json` 외에 CSV, GeoJSON, XLSX가 생성되면 안 된다.
- `npm run check:safety`가 통과해야 한다.
- 추가 수동 검색:
  - `B000` 검색 결과가 없어야 한다.
  - 실제 학교명 패턴 검색 결과가 없어야 한다.
  - 실제 좌표 패턴 검색 결과가 없어야 한다.
- 표본 학교 3개 이상을 골라 후보지/공원/단지/구역 라벨이 `A/B/C` 형태인지 확인한다.
- 상대좌표가 실제 위도/경도가 아니라 `x_m`, `y_m` 같은 학교 중심 좌표인지 확인한다.

## 단계 3. 첫 화면과 앱 요약 복원

목표:
- 기존 앱의 안내/요약 흐름을 최대한 살린다.

작업:
- 첫 화면에 프로젝트 메시지, 핵심 흐름, 정책지원 원칙을 배치한다.
- 앱 요약 모달 또는 요약 페이지를 만든다.
- 기존 앱의 “반경 너머, 도달 가능성” 메시지는 유지하되 식별 예시는 쓰지 않는다.

완료 기준:
- 사용자가 첫 화면에서 앱 취지와 주요 기능을 이해하고 지도 화면으로 진입할 수 있다.

성공 검증:
- `npm run build`가 통과해야 한다.
- Playwright 또는 브라우저로 첫 화면이 열려야 한다.
- 첫 화면에서 지도 화면으로 이동하는 버튼이 작동해야 한다.
- 첫 화면 문구에 원본 학교명, 특정 공원명, 특정 단지명, 실제 주소가 없어야 한다.
- 콘솔 에러가 없어야 한다.
- 모바일 폭과 데스크톱 폭에서 주요 텍스트가 겹치지 않아야 한다.

## 단계 4. 비식별 지도 화면 구현

목표:
- 기존 지도 화면의 탐색 경험을 실제 지도 없이 재현한다.

작업:
- 전체 보기에서는 구별/유형별 비식별 학교 점을 보여준다.
- 줌인 또는 학교 선택 시 학교별 도식지도로 전환한다.
- 학교별 도식지도에는 다음만 표시한다:
  - 학교 중심점
  - 반경 500m 원
  - 도보 500m 범위
  - 공원 A/B/C
  - 후보지 A/B/C
  - 단절요소 A/B/C
- 도로명, 역명, 공원명, 단지명, 실제 타일 배경은 표시하지 않는다.

완료 기준:
- 지도 화면에서 식별 가능한 실제 지리 정보가 없다.
- 기존 앱처럼 학교 선택 후 상세 리포트로 이어진다.

성공 검증:
- 실제 지도 SDK, 카카오맵, 외부 지도 타일 네트워크 요청이 없어야 한다.
- 지도 화면에 실제 도로명, 역명, 공원명, 단지명, 주소가 표시되지 않아야 한다.
- 전체 보기에서 비식별 학교 점/목록을 선택할 수 있어야 한다.
- 학교 선택 후 500m 반경과 도보 500m 범위가 도식으로 보여야 한다.
- 학교별 도식지도에 학교, 공원 A/B/C, 후보지 A/B/C, 단절요소 A/B/C가 상대 위치로 표시되어야 한다.
- 줌인 또는 학교 선택 흐름 이후 상세 리포트로 이동할 수 있어야 한다.
- `npm run check:safety`가 통과해야 한다.
- Playwright 스크린샷으로 전체 보기와 학교별 도식지도 화면을 각각 저장해 확인한다.

## 단계 5. 상세 리포트 복원

목표:
- 기존 상세 리포트의 정책 판단 흐름을 비식별 상태로 되살린다.

작업:
- 현재 격차, 미래 수요, 접근 마찰, KNN 비교, 정책 해석 섹션을 구성한다.
- 학교명과 유사학교명은 비식별 코드로만 표시한다.
- 공원명은 `공원 A` 등으로 표시하거나 거리만 표시한다.

완료 기준:
- 기존 상세 리포트와 정보 구조가 유사하다.
- 식별자 없이도 정책 판단 근거가 읽힌다.

성공 검증:
- 상세 리포트에 현재 격차, 미래 수요, 접근 마찰, KNN 비교, 정책 해석 섹션이 모두 있어야 한다.
- 학교명 대신 비식별 학교 코드만 표시되어야 한다.
- 유사학교도 비식별 코드만 표시되어야 한다.
- 공원명은 실제명 없이 거리 또는 `공원 A` 형식이어야 한다.
- 기존 앱의 핵심 지표와 비교해 값 누락이 없는지 표본 학교 3개 이상 확인한다.
- 상세 리포트에서 시뮬레이션으로 이동하는 버튼이 작동해야 한다.
- `npm run build`와 `npm run check:safety`가 모두 통과해야 한다.
- 콘솔 에러가 없어야 한다.

## 단계 6. 시뮬레이션 복원

목표:
- 기존 후보지 시뮬레이션의 필터와 가중치 조정 경험을 살린다.

작업:
- AI 추천 모드와 직접 설정 모드를 구현한다.
- 제외 조건, 가중치, 후보 순위, 후보 상세를 표시한다.
- 후보별 좌표와 원본 grid ID는 숨긴다.
- 후보지 경로는 실제 경로선 대신 상대거리 도식선으로 표시한다.

완료 기준:
- 후보지 추천 결과가 비식별 상태로 비교 가능하다.
- 후보지 A/B/C만으로도 수요, 거리, 단절요소를 비교할 수 있다.

성공 검증:
- AI 추천 모드와 직접 설정 모드가 모두 열려야 한다.
- 제외 조건 토글이 후보 목록에 반영되어야 한다.
- 가중치 조정이 후보 순위 또는 점수에 반영되어야 한다.
- 후보지 이름은 `후보지 A/B/C` 형식이어야 하고 원본 grid ID가 보이면 안 된다.
- 후보 상세에는 수요, 학교 거리, 기존 공원 거리, 단절요소 요약이 표시되어야 한다.
- 실제 좌표, 실제 경로 좌표 배열, 실제 시설명이 화면과 JSON에 없어야 한다.
- 후보지가 없는 학교도 오류 없이 “표시 가능한 후보 없음” 상태를 보여야 한다.
- `npm run build`와 `npm run check:safety`가 모두 통과해야 한다.
- Playwright로 후보 선택, 필터 변경, 가중치 변경 화면을 확인한다.

## 단계 7. 전체 통계 복원

목표:
- 기존 전체 통계 페이지의 구별 비교와 Top 목록을 유지한다.

작업:
- 시 전체 KPI, 구별 우선 지원 압력, 구별 Top 학교, 전체 Top 학교를 구현한다.
- 모든 학교명은 비식별 코드로 표시한다.
- 구 단위는 유지한다.

완료 기준:
- 기존 통계 화면과 같은 의사결정 흐름을 제공한다.

성공 검증:
- 전체 KPI, 구별 우선 지원 압력, 구별 Top 학교, 전체 Top 학교가 표시되어야 한다.
- 학교명은 모두 비식별 코드여야 한다.
- 구 단위 필터/선택이 작동해야 한다.
- Top 학교 클릭 시 해당 비식별 학교 상세로 이동해야 한다.
- 통계 수치가 `statistics_anon.json`과 일치해야 한다.
- 특정 학교를 재식별할 수 있는 원본명, 학교ID, 좌표, 시설명이 없어야 한다.
- `npm run build`와 `npm run check:safety`가 모두 통과해야 한다.

## 단계 8. 최종 검증과 배포

목표:
- 공개 제출 가능한 상태로 잠근다.

작업:
- `npm run build:data`
- `npm run build`
- `npm run check:safety`
- Playwright로 첫 화면, 지도, 상세 리포트, 시뮬레이션, 전체 통계 확인
- 새 GitHub 저장소에만 push
- 별도 Vercel 프로젝트로 배포

완료 기준:
- 기존 앱 URL과 제출 앱 URL이 분리되어 있다.
- 제출 앱 산출물에 원본 식별자가 없다.
- 화면 구조는 기존 앱의 핵심 흐름을 유지한다.

성공 검증:
- `npm run build:data`가 통과해야 한다.
- `npm run build`가 통과해야 한다.
- `npm run check:safety`가 통과해야 한다.
- `git status --short`에서 커밋 대상이 제출용 파일뿐인지 확인한다.
- `git ls-files`에 `xlsx`, `csv`, `geojson`, `data_processed`, `.env`가 없어야 한다.
- `dist/` 산출물에서도 식별자 스캔이 통과해야 한다.
- Playwright로 다음 화면을 모두 확인한다:
  - 첫 화면
  - 전체 지도/도식지도
  - 학교 상세 리포트
  - 후보지 시뮬레이션
  - 전체 통계
  - 앱 요약/가이드
- 브라우저 콘솔에 에러가 없어야 한다.
- 기존 앱 URL과 제출 앱 URL이 다르고, 제출 앱이 기존 앱의 `data_processed` 경로를 요청하지 않아야 한다.
